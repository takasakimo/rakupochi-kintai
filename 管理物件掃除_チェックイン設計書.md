# 管理物件掃除 - チェックイン/チェックアウト機能 設計書

## 1. 概要

らくポチ勤怠の自社向け大幅リニューアルの一環として、営業先入店報告と同様に**管理物件の掃除案件**向けのチェックイン/チェックアウト機能を新規作成する。

### 1.1 背景・目的

- 管理物件の掃除案件が開始されるため、現場スタッフの打刻・作業報告・引き継ぎを一元管理する
- 1日に最大10ヶ所以上を回る運用を想定し、**1日に何度も打刻可能**なシステムとする

### 1.2 スコープ

| 項目 | 内容 |
|------|------|
| 対象 | 管理物件の掃除作業 |
| ユーザー | 現場スタッフ（打刻側）、管理者（アサインメント・確認側） |
| 打刻回数 | 1日あたり複数回（最大10ヶ所以上を想定） |

### 1.3 機能のON/OFF制御

- このチェックイン/チェックアウト機能は**スーパー管理者**によってON/OFFの切り替えが可能とする
- 企業（テナント）単位で有効/無効を設定する想定

---

## 2. 基本フロー

### 2.1 1件あたりの流れ

```
スタッフが物件へ到着
    ↓
【チェックイン】打刻 + 写真 + 場所 + 引き継ぎ確認（あれば）
    ↓
作業実施
    ↓
【チェックアウト】打刻 + 写真 + 作業完了報告
    ↓
次の物件へ
```

---

## 3. チェックイン仕様

### 3.1 必須項目

| 項目 | 内容 |
|------|------|
| チェックイン打刻 | 到着時刻を記録 |
| 作業開始前の建物外観写真 | 1枚（建物全体＋貸与時計を写す） |
| 作業場所 | GPS自動取得（住所・緯度経度） |

### 3.2 写真の撮影ルール（チェックイン）

- **枚数**: 1枚固定
- **撮影内容**: 建物全体＋貸与する時計を同一フレームに写す

### 3.3 写真撮影の操作とメタデータ（チェックイン・チェックアウト共通）

| 項目 | 仕様 |
|------|------|
| **操作** | 写真撮影ボタンを押してカメラを起動する |
| **メタデータ** | 撮影日時・GPS（緯度経度）を自動付与（改ざん防止・証跡として） |
| **位置情報オフ時** | 写真へのGPS付与は行わない（打刻時のGPSで補完）。エラー表示は出さず、そのまま保存する |

### 3.4 引き継ぎ事項がある場合

- 前回の作業者からの引き継ぎ事項がDBに存在する場合：
  1. チェックイン打刻と写真撮影を提出
  2. 自動的に引き継ぎ事項を表示
  3. 確認完了ボタンを押す
  4. 打刻を完了

### 3.5 チェックインとチェックアウトの紐付け

- 1件の「作業記録」＝チェックイン1件 + チェックアウト1件でペア管理
- 未完了（チェックアウト忘れ）を一覧で検知可能にする（アラート通知は実装しない）

---

## 4. チェックアウト仕様

### 4.1 必須項目

| 項目 | 内容 |
|------|------|
| チェックアウト打刻 | 退去時刻を記録（GPS自動取得） |
| 作業完了写真 | 外観1枚＋ゴミ袋（複数枚可） |
| 作業完了報告 | 以下の3要素を入力 |
| 報告完了ボタン | 押下で打刻完了 |

### 4.2 写真の撮影ルール（チェックアウト）

- **外観**: 1枚固定（建物全体＋貸与時計を写す）
- **ゴミ袋**: 複数枚可能（建物全体＋貸与時計を写す）
- **共通**: すべての写真に建物全体と貸与する時計を同一フレームに含める（時刻の証跡・改ざん防止）

### 4.3 作業完了報告の内容

| 項目 | 説明 |
|------|------|
| 所感 | 作業全般の感想・メモ |
| 特に汚れていた場所 | 今後の注意事項として記録 |
| 次回への引き継ぎ事項 | 次回の作業者向けメモ |

## 5. 追加機能

### 5.1 所要時間の自動記録（必須）

- チェックイン〜チェックアウトの時間を自動計算して保存
- 物件ごとの作業時間・1日の稼働時間の集計に活用

### 5.2 作業種別の選択（必須）

スタッフがチェックアウト時に作業種別をプルダウンで選択して記録する。

| 種別 | 初期選択 | 備考 |
|------|----------|------|
| 定期清掃 | **初期選択**（デフォルト） | |
| 追加清掃 | | |
| 特別清掃 | | |
| その他 | | 選択時のみ**コメント入力欄**を表示し、任意の内容を追記可能 |

### 5.3 管理者側：スタッフへの導線・アサインメント

| 機能 | 内容 |
|------|------|
| 当日の訪問順の指定 | 管理者が「①物件X → ②物件Y → ③物件Z」のように順番を指定 |
| 物件ごとの地図リンク | 登録住所から Google マップ / Apple マップ へのリンクを自動生成 |
| 1日のアサインメント | 今日の担当物件をスタッフに割り当て、スタッフは自分の分だけ表示 |
| 地図上のルート表示 | 訪問順のマーカー＋線で巡回ルートを可視化（既存 Leaflet を流用） |
| ルート提案（任意・将来） | 複数物件の住所から巡回順・ルートを提案 |

### 5.4 スタッフ側のUX（概要）

1. 当日朝に「今日のスケジュール」が届く
2. 順番通り「次は物件X」→ 地図でナビ → 到着後にチェックイン
3. 作業 → チェックアウト → 「次は物件Y」へ

### 5.5 スタッフ向け画面：チェックイン/チェックアウトタブ（詳細）

従業員画面に**チェックイン/チェックアウト**タブを新設し、管理者から送信された1日のスケジュールを表示する。

#### 画面構成

```
【3月12日(木)】

[今日の訪問ルート] ← タップで地図表示（訪問順のマーカー＋線）

┌─────┬─────────────────────────────────────────────────────────┐
│  1  │ 物件名　　住所                                            │
│     │ [チェックイン]  [チェックアウト]  [報告内容編集]             │
├─────┼─────────────────────────────────────────────────────────┤
│  2  │ 物件名　　住所                                            │
│     │ [チェックイン]  [チェックアウト]  [報告内容編集]             │
├─────┼─────────────────────────────────────────────────────────┤
│  3  │ … （案件がない枠は非表示）                                  │
└─────┴─────────────────────────────────────────────────────────┘
```

#### 動作仕様

| 要素 | 動作 |
|------|------|
| **今日の訪問ルート** | タップで地図モーダルを表示。訪問順（①→②→③…）のマーカーと線でルートを可視化 |
| **物件名** | タップで住所と地図を表示（該当物件の位置を地図上に表示） |
| **チェックイン欄** | 未打刻時は打刻ボタン、打刻済みは時刻を表示。タップ→確認ダイアログ→打刻フロー（写真・GPS・引き継ぎ確認） |
| **チェックアウト欄** | 未打刻時は打刻ボタン、打刻済みは時刻を表示、**完了時は「済」を表示**。タップ→確認ダイアログ→打刻フロー（写真・作業完了報告） |
| **誤タッチ防止** | チェックイン・チェックアウトボタン押下時に確認ダイアログを表示（例：「○○物件にチェックインしますか？」）。[キャンセル] [チェックイン] で意図を確認してから打刻フローへ進む |
| **完了後** | チェックアウトが完了した物件は、チェックイン・チェックアウトともに**押せなくする（打刻不可）**。修正が必要な場合は申請フローを利用 |
| **報告内容編集** | 写真追加、コメント（所感・汚れ箇所・引き継ぎ）の追加・修正が可能。**作業当日、日付が変わるまで**のみ編集可。打刻時刻の変更は不可 |
| **空き枠** | アサインメントがない番号（例：4〜10が未割当の場合）は行ごと非表示 |

#### データフロー

- 管理者が1日のアサインメント（物件＋訪問順）を作成・送信
- スタッフは自分の担当分のみ表示
- チェックイン・チェックアウトはその場で打刻（申請フローは経由しない）

### 5.6 管理者による報告確認

- **閲覧のみ（承認フローなし）** … スタッフの報告送信時点で完了とする。管理者は一覧・詳細を閲覧するだけ
- **問題がある場合** … 管理者からスタッフへ**直接連絡**で対応（システム上の差し戻しは行わない）

### 5.7 管理者側の画面・操作（構成案）

管理者が利用する画面の構成と主な操作。

#### 5.7.1 物件マスタ管理画面

| 機能 | 内容 |
|------|------|
| 物件一覧 | 登録済み物件の一覧表示（物件名、住所、施錠有無など） |
| 物件登録 | 新規物件の登録。住所入力＋地図上でピン差し（セクション6.4参照） |
| **一括登録** | CSVインポート、テンプレート複製（セクション6.5参照） |
| 物件編集 | 既存物件の情報変更。住所・ピン位置の調整可 |
| 地図表示 | 一覧から物件を選択して地図で位置確認、または登録・編集時にピン配置・ドラッグ調整 |

#### 5.7.2 アサインメント作成画面

**作成モード（2パターン対応）**

| モード | 操作イメージ |
|------|--------------|
| **人員ごとに案件を指定** | スタッフを選ぶ → 物件を選んで順番を決める → 次のスタッフへ… |
| **案件に人を当て込む** | 物件を選ぶ → 担当スタッフを割り当てる → 該当スタッフの訪問順に組み込む |

**共通機能**

| 機能 | 内容 |
|------|------|
| 対象日選択 | アサインメントを作成する日付を指定 |
| 物件追加・順番 | マスタから物件を選択し、訪問順（①→②→③…）で追加。ドラッグ＆ドロップで順番変更 |
| 物件ソート | **市区町村ごとのソート** … 住所から市区町村を抽出し、物件を市区町村順に表示 |
| 物件ソート | **当月中に回れていない物件** … 物件マスタの「最終訪問日」が当月中でない物件を優先表示・フィルタ（セクション6.6参照） |
| 地図プレビュー | 訪問ルートを地図上にマーカー＋線で表示（スタッフに表示するルートと同じ） |
| 送信・共有 | 確定後にスタッフへ案件を共有。スタッフのチェックイン/チェックアウトタブに表示される |

#### 5.7.3 進捗・作業記録一覧画面

| 機能 | 内容 |
|------|------|
| 日付・スタッフ・物件で絞り込み | 指定日の全スタッフ、または特定スタッフ・特定物件の作業状況を表示。履歴として過去分も検索可能 |

| 地図表示 | 打刻時の位置を地図上で確認（既存のLeaflet地図機能を流用） |

#### 5.7.4 スタッフ向け履歴閲覧

| 対象 | 内容 |
|------|------|
| **閲覧者** | スタッフ本人のみ（自分の作業記録のみ表示） |
| **絞り込み** | 日付指定で自分の作業記録を一覧表示 |
| **詳細** | 完了した物件の詳細（写真・報告・引き継ぎ事項）を確認可能 |
| **閲覧期間** | MVP段階では特段の制限なし。運用を見ながら検討 |

## 6. 物件マスタ・案件共有

### 6.1 データの流れ

1. **マスタの受け取り** … 物件マスタ情報が外部から提供（卸）される
2. **管理者がマスタ登録** … 受け取った物件情報をシステムに登録
3. **案件共有** … 管理者がスタッフに案件（アサインメント）を共有
4. **スタッフに表示** … 共有された案件について、登録済みの物件情報が表示される

### 6.2 物件マスタ登録項目（管理者が登録）

案件共有時にスタッフへ表示される物件情報。管理者がマスタ登録時に入力する。

| 項目 | 必須 | 備考 |
|------|------|------|
| 物件名 | ○ | |
| 住所 | ○ | 地図リンク生成、ルート表示に使用 |
| 緯度・経度 | ○ | 住所入力＋ピン差しで登録（セクション6.4参照） |
| 共有部分の施錠有無 | ○ | オートロック解除キー情報などを含む |
| 管理人の有無 | ○ | 報告の必要有無に連動 |
| 駐車可能スペースの有無 | ○ | なしの場合は近隣コインパーキングの場所・金額を記載 |
| 鍵・アクセス方法 | △ | 施錠有無の補足（キーボックス、宅配ボックス等） |
| 連絡先 | △ | 管理人・緊急連絡先 |
| 作業範囲・注意事項 | △ | 自由記述 |
| 建物の入り方 | △ | インターホン番号、玄関・通用口の違いなど |
| 最終訪問日 | 自動 | システムで自動更新（セクション6.6参照）。管理者は登録しない |

※ △＝あれば便利（任意）

### 6.3 案件共有とセキュリティ

- **案件共有時**: 上記の物件情報がスタッフに表示され、作業に利用可能
- **案件終了後**: **スタッフからは物件情報を閲覧できなくする**（施錠・連絡先等のセキュリティ情報を含むため）

### 6.4 物件の位置情報の登録方法（住所＋ピン差し）

ジオコーディングで出ない物件にも対応するため、**住所入力とピン差しの両方**をサポートする。

| 方法 | 流れ | 用途 |
|------|------|------|
| 住所入力 | 住所を入力 → ジオコーディングで緯度経度取得 → マーカー表示 | 一般的な物件 |
| ピン差し | 地図上をクリックしてマーカー配置、またはドラッグで位置調整 | 住所が出ない物件・位置の微調整 |
| 併用 | 住所入力で大まかな位置表示 → 必要ならドラッグで微調整 | 推奨フロー |

**想定されるケース**: 新築・ビル名のみ、郊外、合併で住所表記が変わった物件など、ジオコーディングでヒットしない場合にピン差しで直接座標を指定する。

### 6.5 物件の一括登録・効率化

1件ずつの登録負荷を減らすため、以下の機能を用意する。

| 機能 | 内容 |
|------|------|
| **CSV一括インポート** | 物件名・住所などをCSVで用意してアップロード。住所からジオコーディングを一括実行し、一括で登録する。マスタ卸しデータをそのまま使える想定 |
| **テンプレート複製** | 既存の物件をコピーして新規作成。物件名・住所などを差し替え、他は流用して微調整する |
| **失敗・ずれの補正** | ジオコーディング失敗や位置ずれがある物件は、一括登録後に一覧からピン差しで個別に修正する |
| **段階的登録** | 初回は最小限（物件名・住所）のみ登録し、施錠・駐車・連絡先などは案件共有前に補完することも可能とする |

### 6.6 最終訪問日と自動更新

- **最終訪問日** … 物件マスタに保有。その物件でスタッフがチェックアウト打刻を完了した日時を記録する
- **自動更新のタイミング** … スタッフが案件のチェックアウト打刻を完了したとき、その物件の「最終訪問日」をチェックアウト日時で自動更新する
- **利用用途** … アサインメント作成時の「当月中に回れていない物件」の判定に使用。最終訪問日が当月中でない物件（未訪問・先月以前）を優先表示・フィルタする

## 8. 打刻漏れ・打刻修正の申請

既存の**申請機能**（打刻修正、残業、休暇、立替金、交通費、シフト希望）を拡張し、**チェックイン/チェックアウト漏れの申請**を追加する。

| 項目 | 内容 |
|------|------|
| 申請タイプ | `cleaning_check_omission`（新規） |
| 申請元 | 既存の申請画面（従業員・管理者ともに利用可能） |
| フロー | 従業員が申請 → 管理者が承認/却下 → 承認時に作業記録を補完 |
| 申請内容（content） | 物件ID、日付、漏れ種別（チェックイン/チェックアウト）、想定時刻、理由 など |

通常の打刻修正（`attendance_correction`）とは別に、掃除案件専用の申請タイプとして管理する。

## 10. 実装・デプロイの必須ルール

本機能の実装にあたり、以下のルールを**必ず遵守**する。

| ルール | 内容 |
|--------|------|
| **デプロイの自動化** | 実装した変更は、**Vercel CLI** または CI/CD 等により**自動でデプロイ**すること |
| **エラー確認** | デプロイ後、**エラーが出ていないか確認**することを必ず行う。ビルドエラー・実行時エラー・コンソールエラー等を確認し、問題があれば修正してから次に進む |

**運用方針**: タスクや機能単位で実装を区切り、各区切りごとにデプロイ → エラー確認を実施してから次の作業に進むこと。

## 11. 実装フェーズ

各フェーズ完了ごとに**Vercel CLI でデプロイ**し、**エラーが出ていないか確認**してから次へ進むこと。

### フェーズ1: 基盤・設定

| 項目 | 内容 |
|------|------|
| DBスキーマ追加 | 物件（Property）テーブル、アサインメント（Assignment）テーブル、作業記録（CleaningWorkRecord）テーブル、写真・引き継ぎ等の関連テーブル |

| スーパー管理者ON/OFF | チェックイン/チェックアウト機能の有効/無効を企業単位で切り替え |
| ルーティング・ナビ | 管理者・スタッフのメニューに本機能への導線を追加（機能OFF時は非表示） |

**完了条件**: スキーママイグレーション成功、設定画面でON/OFFが動作する

---

### フェーズ2: 物件マスタ

### フェーズ4: スタッフ側チェックイン/チェックアウト

**重要：打刻時刻の取得**
- チェックイン/チェックアウトの時刻取得は、**勤怠の打刻**（`app/employee/clock/page.tsx`）で使用しているものと**同一**とする
- 既存コードの修正は行わず、新規実装時に同じロジックを参照して実装する

```
// 勤怠打刻で使用中の実装（app/employee/clock/page.tsx 94-100行付近）
const [currentTime, setCurrentTime] = useState(new Date())
useEffect(() => {
  const timer = setInterval(() => setCurrentTime(new Date()), 1000)
  return () => clearInterval(timer)
}, [])

const getCurrentTimeString = () => currentTime.toTimeString().slice(0, 5)  // "HH:MM"
const getCurrentDateString = () => currentTime.toISOString().split('T')[0]  // "YYYY-MM-DD"
```

- APIには `{ time: getCurrentTimeString(), date: getCurrentDateString(), location }` の形式で送信する
- 既存システムを壊さないため、上記の修正・共通化は行わないこと

| 項目 | 内容 |
|------|------|
| チェックイン/チェックアウトタブ | スタッフ向け専用タブ、スケジュール一覧（物件名・住所、チェックイン・チェックアウト・報告内容編集） |
| チェックインフロー | 打刻、写真1枚（建物全体＋貸与時計）、GPS取得、引き継ぎ確認（あれば）、確認ダイアログ |
| チェックアウトフロー | 打刻、外観1枚＋ゴミ袋複数、作業完了報告（所感・汚れ箇所・引き継ぎ）、作業種別選択、確認ダイアログ |
| 写真保存 | Supabase Storage 等へのアップロード、撮影日時・GPSメタデータ付与（位置オフ時はスキップ） |
| 報告内容編集 | 作業当日・日付が変わるまで、写真追加・コメント修正のみ可能 |
| 今日の訪問ルート | 地図上にルート表示 |
| 誤タッチ防止 | チェックイン・チェックアウト前に確認ダイアログ表示 |
| 完了後の扱い | チェックアウト完了時は「済」表示、打刻不可。報告内容編集は当日のみ可 |

**完了条件**: スタッフがチェックイン→作業→チェックアウトの一連の流れを完了できる
